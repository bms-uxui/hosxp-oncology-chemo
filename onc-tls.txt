1) Goals & Non-Goals
1.1 Goals

สั่งยาเคมีบำบัดแบบ Regimen/Cycle-based ภายใน HIS (ไม่แยกระบบ)

ใช้ข้อมูลเดิมของ HOSxP: HN, VN/AN, drug master, lab, billing, inventory

เพิ่มความปลอดภัยด้วย Clinical Rules + Lab Gate + PIN/e-sign

รองรับ workflow จริง: Doctor → Pharmacist verify → Compounding → Administer

ติดตามสถานะ/รายงาน/แดชบอร์ด

1.2 Non-Goals (รอบแรก)

ไม่ทำ AI recommendation ใน Phase 1

ไม่ทำเชื่อมระบบภายนอกทุกชนิด (ยกเว้นจำเป็นตาม site)

ไม่ทำ mobile app ใหม่ (แต่ UI ต้อง responsive)

2) Users, Roles, Permissions (RBAC)
2.1 Roles

ONC_DOCTOR: สร้าง/แก้ regimen plan, สั่งยา, ยกเลิก, sign

ONC_PHARMACIST: ตรวจสอบคำสั่ง, ปรับ dose ตาม rule/แนะนำ, approve/verify

COMPOUND_TECH: เตรียมผสมยา, บันทึก lot/expiry/waste, สถานะ prepared

CHEMO_NURSE: บันทึกการให้ยา/เวลาจริง/อาการไม่พึงประสงค์, complete administration

BILLING_OFFICER: ตรวจ charge, package mapping, export รายการ

ADMIN: ตั้งค่า master/regimen/rules/thresholds/permissions

2.2 Permission Matrix (ตัวอย่าง)

Doctor: create/submit/cancel order, view all, edit plan before verify

Pharmacist: verify/reject, edit dose (ตาม policy), lock plan after verify

Nurse: administer only verified+prepared

Admin: manage masters & rules, not allowed to sign clinical order

6) Key Business Rules
6.1 Safety Gate (Hard Stop)

ไม่มี diagnosis (ICD-10) ใน VN/AN → ห้าม submit chemo order

ไม่มี BSA/Weight ล่าสุดตาม policy → ห้าม submit

LAB_GATE: ANC/PLT/CrCl/LFT ต่ำ/สูงเกิน threshold → BLOCK หรือ REQUIRE_OVERRIDE ตาม rule

แพ้ยา (allergy) ตรงกับ drug_code → BLOCK หรือ override ตามสิทธิ์

หลัง Pharmacist Verify แล้ว:

Doctor แก้ไข dose ได้เฉพาะ “แก้เป็น revision ใหม่” (สร้าง order เวอร์ชันใหม่) เพื่อรักษา auditability

6.2 Dose Calculation (Core)

dose_calc_method:

BSA: dose = base_dose * BSA

WEIGHT: dose = base_dose * weight_kg

AUC: (ถ้าต้องใช้) คำนวณตามสูตรที่ตั้งค่า (ต้องระบุใน regimen)

FIXED: ใช้ planned_dose

Dose rounding:

policy เช่น round to nearest X mg หรือ vial-based rounding (กำหนดได้ใน regimen หรือ global)

6.3 Workflow Status Transition

DRAFT → SUBMITTED (Doctor + PIN)

SUBMITTED → VERIFIED (Pharmacist + PIN/Sign)

VERIFIED → PREPARED (Compounding)

PREPARED → ADMINISTERED (Nurse)

Any state → CANCELLED (ตามสิทธิ์ + PIN + reason required)

HOLD:

day/status = HELD หาก lab fail หรือแพทย์สั่ง hold

9) Security & Audit
9.1 Authentication

ใช้ SSO/HOSxP auth token เดิม

ทุก request ต้องมี user_id และ role ใน token/claims

9.2 PIN / e-Sign

PIN ต้อง:

เก็บแบบ hashed + salt

rate limit attempts

require reset workflow

Actions requiring PIN:

submit / verify / cancel / override hard-stop

9.3 Audit Log

ต้องบันทึกอย่างน้อย:

entity, action, before/after, user, timestamp, ip, device (ถ้ามี)

ห้ามแก้ไข audit log (append-only)

10) Non-Functional Requirements
10.1 Performance

Worklist load < 2s สำหรับ 500 รายการ/วัน

Rule evaluate < 300ms ต่อ order (target)

10.2 Availability

99.5% ในเวลาราชการ (หรือ site policy)

มี graceful degradation ถ้า lab service ล่ม (อ่านค่าล่าสุด + เตือน)

10.3 Data Integrity

ใช้ transaction ตอน submit/verify

optimistic locking สำหรับ order revision
